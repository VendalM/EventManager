# Менеджер событий (EventManager)

Простое и понятное API для управления вашими событиями. Создавайте, редактируйте и удаляйте события с помощью удобных HTTP-запросов.

## Что умеет это API?

### События

*   **Посмотреть все события** — `GET /events`
*   **Найти событие по номеру** — `GET /events/{id}`
*   **Добавить новое событие** — `POST /events`
*   **Исправить событие** — `PUT /events/{id}`
*   **Удалить событие** — `DELETE /events/{id}`

Событие возвращается в следующем формате:

```json
{
  "id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "title": "Конференция .NET",
  "description": "Ежегодная конференция .NET разработчиков",
  "startDate": "2024-12-15T10:00:00Z",
  "endDate": "2024-12-15T18:00:00Z",
  "totalSeats": 100,
  "availableSeats": 42
}
```

**Как работает:** При создании события указывается общее количество мест (`TotalSeats`). Количество свободных мест (`AvailableSeats`) изначально равно общему количеству мест. При успешном бронировании `AvailableSeats` уменьшается на 1, при отмене бронирования — восстанавливается.

| HTTP статус | Описание |
|-------------|----------|
| 200 OK | Запрос выполнен успешно (GET) |
| 201 Created | Событие успешно создано (POST, PUT) |
| 204 No Content | Событие успешно удалено (DELETE) |
| 400 Bad Request | Ошибка валидации данных |
| 404 Not Found | Событие с указанным ID не найдено |
| 409 Conflict | Недостаточно мест при изменении вместимости события |

### Бронирования
- `POST /events/{id}/book` — создать бронь (статус `Pending`)
- `GET /bookings/{id}` — получить информацию о брони

**Как работает:** Бронь создаётся со статусом `Pending`. При создании брони система проверяет наличие свободных мест (`AvailableSeats`). Если места есть, количество доступных мест уменьшается на 1, и бронь создаётся. API возвращает `202 Accepted`, тело брони и заголовок `Location: /bookings/{bookingId}`. Затем фоновый сервис обрабатывает бронь и переводит её в финальный статус (`Confirmed` или `Rejected`). Если мест нет, API возвращает ошибку `409 Conflict`.

| HTTP статус | Описание |
|-------------|----------|
| 202 Accepted | Бронь принята в обработку |
| 200 OK | Бронь получена по ID |
| 404 Not Found | Событие/бронь с указанным ID не найдено |
| 409 Conflict | Нет свободных мест для бронирования |

### Расширенные возможности

*   **Фильтрация событий** — поиск по названию (регистронезависимый) и диапазону дат
*   **Пагинация** — удобная навигация по большому списку событий
*   **Глобальная обработка ошибок** — единообразные JSON-ответы с информацией о проблеме
*   **Защита от овербукинга** — предотвращение превышения количества доступных мест при конкурентных запросах
*   **Потокобезопасное бронирование** — использование `SemaphoreSlim` для синхронизации доступа к критической секции
*   **Фоновая обработка бронирований** — Hosted Service обрабатывает брони со статусом `Pending`

## Как это устроено внутри?

Проект разделен на логические части, чтобы код был чистым и понятным:
*   **Модели (`EventManager.Models`)** — описывают, как выглядит событие и DTO для обмена данными.
*   **Приложение (`EventManager.Application`)** — содержит всю умную логику по работе с событиями (сервисы, интерфейсы).
*   **Презентация (`EventManager.Presentation`)** — отвечает за обработку запросов, Swagger и глобальную обработку ошибок.
*   **Инфраструктура (`EventManager.Infrastructure`)** — доступ к данным через EF Core и PostgreSQL.

### Обработка ошибок

Middleware глобальной обработки исключений перехватывает все необработанные ошибки и возвращает единообразный JSON-ответ:

| Тип ошибки | HTTP статус | Пример ответа |
|------------|-------------|---------------|
| Ошибка валидации | 400 Bad Request | `{"status": 400, "detail": "Не заполнен заголовок события"}` |
| Ресурс не найден | 404 Not Found | `{"status": 404, "detail": "Сущность с идентификатором '3fa85f64-5717-4562-b3fc-2c963f66afa6' не найдена."}` |
| Нет свободных мест | 409 Conflict | `{"status": 409, "detail": "Для события 'Конференция .NET' нет свободных мест."}` |
| Внутренняя ошибка | 500 Internal Server Error | `{"status": 500, "detail": "..."}` |

### Фильтрация событий

Метод `GET /events` поддерживает следующие параметры:

| Параметр | Тип | Описание | Пример |
|----------|-----|----------|--------|
| `title` | string | Поиск по названию (регистронезависимый, частичное совпадение) | `?title=конференция` |
| `from` | DateTime | События, начинающиеся не раньше указанной даты | `?from=2024-12-01` |
| `to` | DateTime | События, заканчивающиеся не позже указанной даты | `?to=2024-12-31` |
| `page` | int | Номер страницы (по умолчанию 1) | `?page=2` |
| `pageSize` | int | Количество элементов на странице (по умолчанию 10) | `?pageSize=20` |

**Пример запроса с фильтрацией и пагинацией:**

`GET /events?title=конференция&from=2024-12-01&to=2024-12-31&page=2&pageSize=10`

## Запуск

Для API используется PostgreSQL. По умолчанию строка подключения берётся из `EventManager/appsettings.json` (`ConnectionStrings:Default`).

```bash
dotnet restore
dotnet run --project EventManager/EventManager.csproj
```

Swagger доступен в Development-среде (по профилю запуска: `https://localhost:7013/swagger` или `http://localhost:5068/swagger`).
